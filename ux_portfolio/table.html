Table 2.0

User testing the next iteration of tables in the xneelo Control Panel.

Executive Summary

This report details a usability study conducted by the UX team regarding the redesign of our tables in the xneelo control panel (also known as Table 2.0). The Table 2.0 prototype was evaluated using the Think Aloud method, which enabled us to examine the user experience and interface and establish any possible usability pain points. These observations were used to refine a list of findings regarding the prototype interface. This report highlights the major findings that currently hamper table usability. The recommendations included here address these issues, and improve upon the user’s overall experience interacting with our tables in the control panel. The five recommendations are to:

  1. Avoid ambiguity by incorporating filters within the search.
  2. Explore row hover states so that they seem clickable.
  3. Don’t allow users to click on the row to select and expand/ collapse the row. Allow row clicks for navigation actions only.
  4. Don’t hide the search when users are selecting rows
  5. Explore listing the selected rows with the number of selections.
  6. Avoid ambiguity about collapsing the search bar by removing the expand/ collapse feature (making the search bar visible at all times).

These recommendations are designed to improve the usability of our tables in the control panel and to provide an enjoyable user experience for customers.

Introduction

The UX team decided to evaluate the following features of the Table 2.0 prototype:

  • Search and Filter
  • Navigation
  • Bulk Select and Bulk Actions
  • Row Actions
  • Pagination
  • Expand/ Collapse Row

With the use of google meet, we were able to record audio, video and screen activity. We then analyzed the qualitative data provided by the user testing, and drew out findings and recommendations concerning the interface's usability.

Methodology

For our testing session, we chose to test with internal xneeloians for quicker recruitment turnaround. The user’s background varied from Tech Support, Admin and Restructures, Scrum Master, to Admin and Billing.

We performed remote moderated usability tests where moderators guided testers through 10 tasks, instructing them to “think aloud” (i.e walk us through their thought process and actions) as they worked through each task. The tasks were designed to test all of our table 2.0 features. The tester’s interactions with the prototype and their commentary was recorded using google meet. We used system usability scale tests in order for testers to quantitatively rate their impressions of each task.

This method allowed us to gather data that we used to evaluate the table 2.0 prototype. From this evaluation, we were able to extract findings of pain points and ultimately made recommendations to improve the usability of our tables in the control panel.

Finding and Recommendations

Finding 1

Users were not confident about how the filtering worked together with the search.

When performing a search, one of the users was questioning how the filter function worked with the search: “What happens when I click the filter button, do I lose what I’ve typed in?” and “I would expect to see the filters within the search function.” Although users would expect that if they perform a search and interact with the filter, the search results would be narrowed down by both criteria, there was ambiguity since both the search and the filter are designed as independent functions.

The search bar is separated from the filter button. Clicking on the search icon button, expands the search bar. When users click on the filter button, an overlay modal appears.

Recommendation 1

Avoid ambiguity by incorporating filters within the search.

Incorporating filters within the search bar will further clarify that the search criteria will not be removed if a filter is applied.

Combine the search and filter as one component. Filters are in the dropdown instead of overlay modal, reassuring the user that their search will remain, when they apply filters.

Finding 2

Users did not click on the table rows for navigation, row expansion nor selection.

When asked to select rows, users clicked on the checkbox. When asked to expand and collapse rows, users clicked on the dropdown icon button. When asked to navigate the table, users clicked on the chevron icon button instead of clicking on the table row. Users were drawn to the ‘View’ tooltip that appeared once they hovered over the chevron.

Currently the hover state is a grey fill, similar to the background colour.

Recommendation 2

Explore row hover states so that they seem clickable.

We want to encourage users to navigate tables by clicking on the rows. A more prominent hover state could achieve this, so that the rows seem more inviting to click on.

Explore the hover state of a row through shadows, making it more enticing to click on. Remove the chevron icon button but keep the icon as a visual indicator.

Recommendation 2b

Don’t allow users to click on the row to select and expand/ collapse the row. Allow row clicks for navigation actions only.

On the Table 2.0 prototype, we allowed users to click on the row to perform a variety of different actions such as navigate to a different screen, select a row and expand/ collapse the row. Only one of the users performed a row click and it was to navigate to a different screen. The same user questioned if clicking on the chevron icon button or clicking on the row performed different actions. (On the Table 2.0 prototype, we allow users to click on either the icon buttons or the rows to perform the same action, causing some uncertainty).

When asked to select a row, all of the participants clicked on the checkbox. On the task that prompted the user to expand/ collapse the row, all of the users clicked on the dropdown arrow. In order to follow the heuristic principle of Consistency and standards, our recommendation is to allow users to navigate through row clicks and not allow users to perform different tasks while doing the same action: “Users should not have to wonder whether different words, situations, or actions mean the same thing. Follow platform and industry conventions.” - NN Group

Finding 3

Users want to search and select at the same time.

We prompted the users to select rows across multiple pages of the table. Users had no issue navigating across multiple pages with pagination. One of the user’s first instinct was to search for the item that was not visible on the table. When he noticed that the search feature was hidden, he then opted to click on the table pagination.

On the default state, you can see the search and filter functions. Once you select one or more rows, the search and filter function are hidden and replaced by text stating the number of selected items.

Recommendation 3

Don’t hide the search when users are selecting rows.

When there is a specific item the user wants to select and there are hundreds of table rows, the user will need to perform a search instead of going through the table pages with the pagination feature. For that reason, we should allow the user to search and select at the same time.

The recommendation is that once one or more rows are selected, we keep the search and the filter functions and rather display the number of selected items and the bulk actions below the search (as opposed to replacing the search and filters).

Finding 4

Users immediate thought when asked to close the search bar was to go back or click on the search icon.

Upon being prompted to close the search bar, users were not immediately confident to click on the screen (outside the search). However, after a couple of tries, they did explore clicking on the screen. One of the users first tried to click on the back button and commented: “Users have become so accustomed to clicking back if they want to reverse what they’ve done.” A couple of users tried clicking on the search icon to close the search bar. This could be because they had to first click on the icon in order to expand the search bar.

Recommendation 4

Avoid ambiguity about collapsing the search bar by removing the expand / collapse feature, making the search bar visible at all times.

This would also eliminate the extra step that the user would need to perform in order to perform a search.

Finding 5

Users want to re-confirm their selection across multiple pages before proceeding with bulk actions.

Even though users noticed the number of selected items in the table header, when they selected rows across multiple pages, they still wanted to check that the selections remained by going back with pagination. Users asked “Would I be able to go to the next page while keeping the [select] criteria?” and “How do I know it’s still there [selected] without proceeding [with the bulk action]?”. Another user said: “In most cases when you select and go to the next page it clears the search selection so it’s very nice that it does that [keep the selection across multiple pages] but I wasn’ 100% it was going to do that.”

When the user selects 1 or more rows, we display the number of rows selected on the toolbar header.

Recommendation 5

Explore listing the selected rows with the number of selections.

Users are not confident to proceed with bulk actions before they are 100% sure of their selections. Listing the selected rows by the number of selected rows, would enable the users to have that visibility at all times and across multiple pages.

Include a dropdown next to the number of selected. The drop down would display a list of all the selections. If the user navigates across table pages, the dropdown and number of selections would be fixed and visible at all times (for as long as 1 or more rows are selected).

Conclusion

With the new interface and experience of our tables, our control panel customers will have a much better experience. We will be able to create consistency and standards in our tables across the application and follow industry conventions. We arrived at these recommendations by performing 6 remote, moderated usability tests and evaluating qualitative data deduced from those tests. The control panel is expected to be improved significantly by adopting these recommendations and launching Table 2.0
